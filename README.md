# Food Order System (外賣點餐與商家管理系統)

## 📌 專案自述與個人背景

本專案是我在退伍前後，為了將大學所學的資料庫理論與業界後端架構接軌，所進行的**個人自主進階實戰專案**。在學校做過幾次簡單的網頁作業後，我深知業界系統不會只有單純的 CRUD（增刪查改）。因此，我參考了成熟的外送平台業務閉環，**一邊理解商務邏輯、一邊手敲程式碼並獨自完成所有 Bug 的除錯**。這個專案幫助我跳脫了學生的開發思維，開始學習如何處理快取、即時通訊與系統分層。

---

## 🛠️ 開發環境與技術棧 (Tech Stack)

為了模擬業界真實開發場景，本專案使用了以下技術與工具進行建構：

*   **後端核心框架**：Java 8 / Spring Boot (2.7.x) / Spring MVC
*   **專案建置與管理**：Maven
*   **資料儲存與 ORM**：MySQL 8.0 / MyBatis 
*   **快取與記憶體資料庫**：Redis
*   **安全與驗證機制**：JWT (JSON Web Token) / ThreadLocal (執行緒安全) / Spring Interceptor (攔截器)
*   **即時通訊協定**：WebSocket
*   **第三方工具與套件**：
    *   **Apache POI**：用於 Excel 營運報表動態產出
    *   **Swagger / Knife4j**：用於自動生成與測試 RESTful API 文件
*   **開發工具**：IntelliJ IDEA / Git / Apifox (API 測試) / DataGrip

## 💡 開發過程與核心收穫

*   **練習業界標準的架構分層 (Spring Boot 三層架構)**
    *   **我是怎麼做的**：嚴格遵循 Controller、Service、Mapper 的分層規範，並使用 Maven 進行相依性管理。
    *   **學到了什麼**：以前寫學校作業程式碼常常亂塞，這次真正體會到「高內聚、低耦合」的好處。配合 Swagger (Knife4j) 產生 API 文件，也讓我更清楚如何設計出格式清晰的 **RESTful API**。
*   **初次導入 Redis 快取處理效能問題**
    *   **我是怎麼做的**：考慮到外送平台的「菜品和套餐分類」是極高頻率被查詢、但很少變動的資料，我將其寫入 Redis 記憶體快取。
    *   **學到了什麼**：我理解到減少資料庫（MySQL）I/O 負載的重要性。同時，我也實作了**快取雙寫一致性**的概念：當商家端修改或刪除菜品時，後端會主動清除對應的 Redis 快取，確保使用者端不會看到過期的選單資訊。
*   **安全機制練習：雙端 JWT 身份驗證**
    *   **我是怎麼做的**：實作自定義的 Spring 攔截器 (Interceptor)，統一攔截非匿名請求並解析 JWT Token。
    *   **學到了什麼**：學會了如何區分管理員與一般用戶的身份。並透過將解析出的用戶資訊存入 **`ThreadLocal`**，確保在同一個請求執行緒中的資料安全，也簡化了後續 Service 層獲取當前登入者資訊的步驟。
*   **整合 WebSocket 實作即時通知**
    *   **我是怎麼做的**：不採用前端定時輪詢（Polling）這種浪費資源的方式，而是整合 WebSocket 協定。
    *   **學到了什麼**：當使用者模擬支付成功後，後端能即時將訊息推播給商家端，觸發語音與彈窗提醒，這讓我完整體驗了**異步即時通訊**在實際商業場景下的應用。
*   **實作動態報表產出 (Apache POI)**
    *   **我是怎麼做的**：在後台營運統計模組中，練習使用 MySQL 聚合函數計算近 30 天的營業額與熱銷排行，並利用 Apache POI 將數據填入 Excel 範本。
    *   **學到了什麼**：掌握了將結構化數據轉化為實體檔案並提供下載的商務功能實作。
*   **實作完整的點餐與訂單狀態機流程 (Order Lifecycle Management)**
    *   **我是怎麼做的**：
        1.  **購物車管理**：設計 `shopping_cart` 邏輯，處理動態加入、修改數量及口味規格關聯。
        2.  **提交訂單與明細分離**：下單時利用單一事務（Transaction）同時寫入 `orders` (主表) 與 `order_detail` (明細表)，並清空購物車，確保資料一致性。
        3.  **模擬支付與狀態演變**：實作狀態機邏輯，嚴格控制訂單從 `待付款 ➔ 待接單 ➔ 派送中 ➔ 已完成` 的商務狀態切換，並在關鍵節點觸發 WebSocket 通知商家。
    *   **學到了什麼**：
        這讓我真正理解了電商/點餐系統核心的業務閉環。我學到了在處理多表操作（訂單主表與明細表）時，必須使用 **`@Transactional` 事務管理**，避免發生「主表建立成功，明細卻漏掉」的嚴重系統 Bug。同時，處理複雜的狀態切換也磨練了我的商務邏輯與邊界條件處理能力。
